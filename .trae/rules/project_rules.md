# Trae 项目规则：AI穿越创业类连载小说

## 1. 项目定位
本项目是长篇连载小说工程，不是一次性短篇。每次写作、续写、改稿、扩纲，都必须优先保证长期一致性、可持续连载性和读者爽感。

小说核心类型：AI穿越创业、程序员创业、科技商业、现实技术推演。

默认主线：废柴程序员张有志从2026年穿越回2016年，凭借对AI技术浪潮的认知，从加密货币赚到第一桶金，创办华天科技，从AI工程化工具切入，用十年时间做大模型基础设施。

## 2. 每次开始工作前必须读取
如果用户要求写正文、续写、扩写、改章、写大纲、调整设定，必须优先读取以下文件：
1. `10_trae控制/01_写前读取清单.md`
2. `10_trae控制/04_防跑偏规则.md`
3. `09_提示词/小说定位.md`
4. `09_提示词/字数节奏规则.md`
5. `00_总设定/总索引.md`
6. 与当前任务直接相关的章节大纲、人物设定、时间线、技术资料。

如果用户只让写下一章，仍然要先确认上一章摘要、当前时间点、人物状态、公司状态和本章目标。

## 3. 写作硬约束
1. 主角可以有未来认知，但不能无成本全知全能。
2. 技术细节要像懂技术的人写的，不要堆术语，不要出现明显伪技术。
3. 创业结果要有因果链：信息差、执行力、资源获取、团队能力、市场窗口共同作用。
4. 爽点必须来自"预判正确 + 提前布局 + 对手误判 + 技术兑现 + 商业成交"，不能只靠旁白宣称主角厉害。
5. 每章必须有明确场景、冲突、推进和尾钩。
6. 避免长篇说明书式科普。技术信息必须嵌入面试、会议、争论、代码排错、客户演示、融资谈判、产品发布等戏剧场景。
7. 不要把现实人物写成主要角色，不要捏造现实公司违法事实。
8. 金融线只能作为启动资金和时代红利，不可喧宾夺主。主线必须回到程序员创业与AI产品。

## 4. 默认文风
爽感优先，废话退散。参考 `第001章_重启.md`：自嘲吐槽与深层孤独交替，克制、冷静、现实质感强。多写动作、物件、环境、沉默、眼神和对话中的信息差。程序员相关场景要写出终端、日志、监控、部署、故障排查和资源约束。主角的内心独白要有"废柴"的自嘲感，但关键时刻要有担当。

## 4.1 行文硬约束（反墨迹规则）
1. **禁止无病呻吟**：不许反复渲染同一种情绪。写孤独写一句就够，不许排比三段。写纠结写一个动作就够，不许内心独白五段。
2. **禁止废话堆砌**：同一个意思不许用不同句式重复表达。"他知道未来"写一次就够了，不许再写"他记得""他了解""他清楚"。
3. **节奏要快**：场景切换要干脆，不许用大段抒情做过渡。读者要的是"接下来呢"，不是"让我再感慨一下"。
4. **结尾要狠**：每章结尾必须干脆利落，一两句话收住。不许排比抒情，不许反复升华，不许"很微弱，但确实存在"这种废话。
5. **内心独白要短**：内心活动最多三句，必须有态度、有信息量。不许写"他不知道该说什么""他想了很久""他忽然觉得眼睛有点酸"这种空转。
6. **信息密度优先**：每一段必须推进剧情或传递新信息。如果删掉某一段不影响上下文，那就必须删掉。
7. **爽感来自行动，不来自感慨**：主角做了一个决定，直接写结果和下一步。不许写"他下定决心""他终于明白了""他眼睛亮了一下"。
8. **少用破折号和省略号做抒情**：破折号用于补充说明，不用于制造文艺感。省略号尽量不用。

## 5. 默认主角
张有志。废柴穿越者。外貌普通到透明，表面自嘲怂萌，内里孤独敏感、重情重义。成长线是从逃避到承担。

## 6. 默认时间线
故事从2016年开始，穿越起点为2016年3月15日。2016年的技术栈以Caffe、TensorFlow 0.x、LSTM、规则引擎为主，不能写出PyTorch（2017年才发布）。

## 7. 文件维护规则
写作产生的新设定必须登记到对应台账：
- 新人物：`02_人物设定/人物总表.md` 或具体人物文件
- 新公司/产品：`03_公司产品线/产品矩阵.md`、`00_总设定/公司状态表.md`
- 新时间节点：`00_总设定/时间线表.md`
- 新伏笔：`00_总设定/伏笔表.md`
- 新技术名词：`00_总设定/技术名词表.md`
- 新章节完成后：`00_总设定/章节摘要表.md`

## 8. 默认禁止项
1. 禁止写成玄幻修仙式科技爽文。
2. 禁止主角随口说出未来产品名就让所有人无条件相信。
3. 禁止团队成员无理由忠诚。
4. 禁止投资人无理由送钱。
5. 禁止客户无理由买单。
6. 禁止用"系统""金手指面板"替代主角判断。
7. 禁止章节只有设定没有剧情。
8. 禁止为了显得专业堆砌不可验证数字。
9. 禁止写成2018年时间线——当前故事从2016年开始。

## 9. 默认交付格式
用户要求写正文时，直接给正文内容。若用户要求"写入文件"，则写入 `07_正文` 中对应章节文件。若用户要求"先给大纲"，则输出本章目标、冲突、场景列表、尾钩。

## 9.1 章节标题规则
1. 标题必须有吸引力，能制造好奇心。禁止用"重启""行业现场""原型"这种抽象概括词。
2. 优先用具体数字、具体物件、具体细节做标题。如"十四万七""四张980Ti""柴犬头像"。
3. 标题要短，2-6个字最佳。
4. 标题要和本章核心场景或核心冲突直接相关。
5. 标题要让读者产生"这是什么？"的疑问，从而点进去看。

## 9.2 技术场景写法规则
1. 正文中禁止使用代码块（```）。代码、命令、报错信息必须用叙事描写表达。
2. 技术细节通过角色动作、对话、内心反应来传递，不要直接贴原始日志。
3. 可以用引号内嵌简短的技术关键词（如"找不到GPU""推理延迟23毫秒"），但不能大段展示。
4. 程序员读者能感受到技术氛围，非程序员读者也能看懂剧情。两者兼顾，优先保证后者能读下去。

## 10. 写完必提交Git
每次写完正文、改完设定、更新大纲后，必须执行：
1. `git add -A`
2. `git commit -m "描述本次改动"`
3. `git push`
不能遗漏，不能攒多次一起提交。
