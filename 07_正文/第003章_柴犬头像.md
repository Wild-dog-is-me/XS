# 第003章 柴犬头像

张有志连续写了四天代码。

窗帘拉着，外卖盒堆在桌角，空调开到最低还是出汗。出租屋的网速只有20M，下载TensorFlow的wheel包等了半个小时。他的笔记本只有8G内存，跑一个简单的模型推理就卡得要死。

但他还是写出了第一个能跑的版本。

功能很简单：用户把一个训练好的TensorFlow模型丢进来，脚本自动完成模型格式转换、依赖打包、生成Docker镜像，然后推送到指定服务器。整个过程一条命令搞定。

```
$ python deploy.py --model saved_model/ --server 192.168.1.100
packing model...
building docker image...
pushing to server...
deploy done. latency: 23ms
```

跑通的那一刻，张有志盯着终端里的`deploy done`看了十秒钟。

第二天他就遇到了问题。

他拿一个真实的图像分类模型测试——Caffe训练的ResNet-50，230MB。打包没问题，Docker镜像构建没问题，推送到服务器也没问题。但服务启动的时候报错了：

```
E tensorflow/core/common_runtime/gpu/gpu_device.cc:334] Could not identify GPU device
```

TensorFlow找不到GPU。

他登录服务器检查，`nvidia-smi`正常输出，CUDA驱动版本8.0，显卡是GTX 980 Ti。但Docker容器里看不到GPU设备。

张有志盯着报错信息想了五分钟，然后反应过来：2016年的Docker默认不支持GPU直通。nvidia-docker还没发布，或者刚发布，用的人很少。他需要在容器里手动挂载NVIDIA的设备文件和驱动库。

他在deploy.py里加了一段逻辑：检测宿主机的NVIDIA驱动版本，自动生成对应的`--device`和`--volume`参数。改完之后重新测试，服务启动成功，推理延迟23ms。

这个问题让他意识到，2016年的AI工程化有多原始。连GPU进容器这种基本操作都得手动处理。在2026年，nvidia-container-toolkit已经是标配，k8s调度GPU资源跟调度CPU一样简单。

但这就是机会。

他知道这个东西有价值。问题是，他一个人做不快。

四天时间，他写了核心的打包和部署逻辑，但Docker镜像构建、GPU环境检测、服务健康检查这些模块还都是空壳。一个人干下去，至少还得两周。

他需要一个帮手。

张有志打开浏览器，在V2EX上发了一个帖子：

**"找一个全栈工程师，做AI模型部署工具，远程兼职，有偿。"**

帖子发出去半天，收到三封私信。前两封一看就是套模板的，第三封让他多看了两眼：

"我是韩洲，94年，全栈，之前在一家外包公司做后台开发。看了你的需求描述，感觉挺有意思的。方便聊聊吗？"

张有志加了他微信。

韩洲的头像是一只柴犬，朋友圈最近一条是转发的"程序员如何保护颈椎"。

"你好，我是韩洲。"对方很快回复。

"你好，张有志。"

"你那个模型部署工具，能详细说说需求吗？"

张有志把huatian-lab的README发了过去。韩洲看了几分钟，回了一段话：

"看明白了。你要做的是一个CLI工具，输入模型文件，输出可部署的Docker镜像。核心难点在环境依赖管理——不同模型可能需要不同版本的CUDA、cuDNN、TensorFlow，这些东西组合起来是个坑。"

张有志盯着这段话看了三秒。

这个人懂行。

不是那种"我会写Python所以我也能做AI"的人，而是真正理解工程化痛点的人。他提到的CUDA/cuDNN版本组合问题，正是张有志这几天写代码时遇到的最头疼的事情。

"你说得对。"张有志回复，"我现在用的是手动指定版本的方式，但扩展性很差。你有什么想法？"

"可以用矩阵配置。"韩洲说，"把CUDA版本、cuDNN版本、框架版本做成一个配置矩阵，用户选一组配置，工具自动生成对应的Dockerfile。这样后面加新框架也方便。"

张有志心里想：这不就是2020年以后各家MLOps平台干的事吗？

嘴上说："这个思路不错，你什么时候有空？"

"晚上和周末。白天还得上班。"

"行，我先把现有的代码推到GitHub上，你看看架构有没有问题。"

"好。"

张有志挂了微信，靠在椅背上。

韩洲。94年，全栈，外包公司。不是什么名校背景，但工程直觉不错。最关键的是——他理解这个问题。

这就是他需要的人。不是算法大牛，不是学术大牛，而是能写代码、能理解工程痛点、能把想法变成可运行工具的人。

他打开终端，把huatian-lab的代码推到了GitHub私有仓库。

```
$ git add -A
$ git commit -m "init: deploy tool v0.1, basic model packaging and push"
$ git push origin main
```

然后他给韩洲发了仓库链接。

"你先看看，有问题随时问。"

"收到。"

张有志关掉微信，看了一眼时间，晚上九点。

他站起来，走到窗边。窗外的城中村灯火通明，有人在楼下吵架，有人在阳台收衣服，有小孩在哭。

他掏出手机，看到苏晚两个小时前发的消息："今天怎么样？吃饭了吗？"

他回了一句："吃了，在忙。"

苏晚秒回了一个猫咪打哈欠的表情包。

张有志笑了一下，把手机收起来。

他回到电脑前，打开了一个新的文件，开始写第二版的需求文档。

写到一半，韩洲发来一条消息：

"刚看完你的代码。打包逻辑写得不错，但有几个问题想问：1.你现在只支持TensorFlow，后面要支持Caffe和PyTorch吗？2.GPU环境检测你是怎么做的？3.部署后的监控有考虑吗？"

张有志看着这三个问题，嘴角抽了一下。

第三个问题：部署后的监控。

这正是他在智语未来看到的痛点。陈远说他们的监控脚本"自己也会挂"。

他回复韩洲："三个问题都有考虑。第一个先不做，2016年TensorFlow是主流，Caffe的部署方式不一样，后面再加。第二个目前是手动指定，后面要做自动检测。第三个——这个很重要，我打算在部署工具里内置一个轻量级的健康检查和指标上报模块。"

"明白了。"韩洲说，"那我先把环境检测这块做了，明天给你看方案。"

"好。"

张有志放下手机，继续写需求文档。

写到凌晨一点，他终于写完了。文档标题是：

**华天Deploy v0.1 产品需求文档**

他把文档推到GitHub，然后关了电脑。

躺在床上，盯着天花板。

韩洲的出现让他意识到一件事：2016年的中国，不缺能写代码的人。缺的是能理解"AI工程化"这个概念的人。大多数程序员还在写CRUD，大多数AI工程师还在调参数。没有人想过"怎么把模型部署到生产环境"这个问题。

但韩洲想到了。

这说明市场上一定还有其他人也想到了。只是他们分散在各个公司里，被淹没在项目制的交付压力中。

张有志需要做的，是把这些分散的人聚起来。

他打开手机，看了一眼苏晚最后发的那个猫咪表情包。

忽然想起一件事。

上辈子，他也在某个深夜写过一个类似的工具。那时候他刚进公司，被分配去解决模型部署的问题。他花了两周写了一个脚本，被leader夸了一句"不错"，然后那个脚本就躺在代码仓库里，再也没人维护过。

那个leader后来跳槽去了字节。

那个脚本后来被另一个新人重写了，因为没人看得懂他的代码。

张有志不想让这一世的huatian-lab也变成那样。

他打开终端，给韩洲发了一条消息：

"明天晚上八点，咱们约个电话，把分工对一下。"

"好。"韩洲秒回。

张有志关掉手机，闭上眼睛。

明天继续写代码。
